## Architecture (browser-backed)

Worker thread owns a persistent headless CloakBrowser; Flask handlers submit
jobs + drain a per-job queue (so streaming crosses threads). Per turn: type
prompt â†’ Enter (frontend mints sentinel/Turnstile/PoW) â†’ capture SSE.

## Mechanism findings (what worked vs not)

- **`window.fetch` hook does NOT work.** Injecting a fetch wrapper via
  `add_init_script` (as an IIFE â€” a bare `() => {}` string never runs, first
  bug) installs fine, but ChatGPT does **not** issue `f/conversation` through
  the page's `window.fetch` (worker/captured ref), so the tee never fires.
- **CDP works.** `Network.streamResourceContent(requestId)` makes
  `Network.dataReceived` deliver the SSE body **incrementally** (verified: 7
  data events over ~2 s) â€” clean live streaming regardless of how the app
  makes the request.
- **Model override via CDP `Fetch`.** Intercept `*/backend-api/f/conversation*`
  at the Request stage and rewrite the JSON body's `model` field, then
  `continueRequest`. Reliable; the anti-bot tokens are headers so they're
  untouched.

## `v1` delta-encoding parser (key correctness bugs fixed)

- **`(o, p)` inheritance:** absent `o`/`p` inherit from the *previous* delta.
  A bare `{"v":{"message":{...}}}` is therefore a new-message **add** (not
  text); a bare `{"v":"text"}` continues the last append. The first parser
  ignored bare dict deltas, so on thinking models the final answer message was
  never registered and its text was misclassified as reasoning.
- **Role gating:** only emit text for `author.role == "assistant"`. Without it,
  the `input_message` echo of the user's own prompt leaked into `content`.
- Reasoning vs answer is decided by the current message's `content_type`
  (`thoughts`/`reasoning_recap`/... â†’ `reasoning_content`; `text` â†’ `content`).

## Known limitations / gotchas

- **Restart contention (not a runtime bug):** killing the server's Python with
  `pkill` while the browser is live makes the CloakBrowser node driver log a
  harmless `EPIPE`; rapid killâ†’relaunch cycles can leave an orphaned Chrome
  holding `.cloak-profile`, so the next launch is flaky. Fix for testing:
  fully kill `openai_proxy` + any `chrome ... .cloak-profile`, wait, then start
  ONE server. During a clean single run it handled many requests + `/health`
  with no crashes. A best-effort `_try_reboot()` relaunches the browser if a
  send throws.
- Serialized (one turn at a time); stateful continuation is a prefix-match
  heuristic; `usage` is zeros; thinking-mini reasoning folds into the answer.
